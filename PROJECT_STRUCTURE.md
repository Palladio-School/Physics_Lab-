# Project Structure

This file maps the current repository before larger documentation, hardware, UI, or performance work. The goal is to make the first cleanup step readable without changing the dashboard implementation.

## Current Product Shape

Physics Lab has two generations in the same repository:

1. Current M5/web-dashboard flow
   - `Physics_Lab/Physics_Lab.ino` runs on the M5StickC Plus2.
   - The M5 creates the `PhysicsLab-M5` Wi-Fi network and exposes the hardware API at `http://192.168.4.1`.
   - `docs/index.html` is the full classroom dashboard, suitable for GitHub Pages or local browser use.

2. Legacy desktop flow
   - `physics_lab_with_angular_velocity.py` is a Python / CustomTkinter app.
   - The firmware still broadcasts UDP packets for compatibility with this app.

New work should state which flow it changes.

## Firmware Responsibilities

The firmware:

- Creates the M5 Wi-Fi access point.
- Reads the active sensor mode.
- Maintains `sampleId`, `sampleMs`, and `sampleRate` so browser polling does not duplicate samples.
- Serves the latest reading and sample batches through HTTP.
- Handles mode changes, sampling-rate changes, and calibration requests.
- Keeps UDP compatibility with the older Python app.

Important endpoints:

- `/`
- `/logo.png`
- `/data`
- `/samples`
- `/set-mode`
- `/settings`
- `/calibrate`

## Dashboard Responsibilities

The full dashboard in `docs/index.html` handles:

- Experiment selection.
- Live graphing and readouts.
- Student-facing worksheet/report interactions.
- CSV export.
- Sampling-rate and calibration controls.
- Digital-only experiments such as the pendulum simulation.
- M5 API polling through the `192.168.4.1` device endpoint when hardware is connected.

Because it is currently a large single HTML file, changes should be narrow and easy to review.

## Hardware Matrix

| Experiment / Mode | Hardware | Firmware Notes |
| --- | --- | --- |
| Acceleration | M5StickC Plus2 IMU | Built-in sensor. |
| Gyroscope | M5StickC Plus2 IMU | Built-in sensor. |
| Force / load cell | HX711 + load cell | Uses `DT`, `DT_ALT`, and `SCK` candidates. |
| Ultrasonic motion | DFRobot URM10 ultrasonic sensor | Uses trigger/echo pins and firmware-side timing. |
| Heat / thermal equilibrium | DS18B20 sensors | Uses two OneWire data pins. |
| Pendulum simulation | No M5 required | Digital dashboard experiment. |

Current pin definitions in `Physics_Lab.ino`:

```cpp
#define TRIG_PIN 26
#define ECHO_PIN 25
#define DT 36
#define DT_ALT 25
#define SCK 26
#define TEMP1_PIN 25
#define TEMP2_PIN 26
```

Firmware startup performs conservative sensor auto-detection. It selects Heat when both DS18B20 sensors are found, Ultrasonic when the URM10 responds to trigger/echo timing, and Force when HX711 is found. Manual selection remains available from the dashboard and M5 buttons.

Pin sharing is intentional in the current firmware mode layout, but it should be documented in a classroom wiring guide before hardware kits are handed out broadly.

## Developer Workflow

Compile firmware:

```bash
cd Physics_Lab
./scripts/arduino.sh compile
```

Upload firmware:

```bash
cd Physics_Lab
./scripts/arduino.sh upload /dev/cu.usbserial-0001
```

Run the legacy app:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python physics_lab_with_angular_velocity.py
```

Inspect the full dashboard locally:

```bash
python3 -m http.server 8000 -d docs
```

Then open:

```text
http://localhost:8000
```

## Known Review Points

- `docs/index.html` should eventually be split into smaller assets/modules, but that should be a separate PR.
- The README and project history should stay aligned so users know whether instructions refer to the current M5/browser flow or the legacy Python flow.
- A hardware/API guide should be the next documentation layer: wiring diagrams, pin choices, calibration steps, expected sensor ranges, and safety notes.
- Polling/performance work should be measured against actual M5 hardware and classroom browser devices.
- GitHub Pages and M5-served firmware pages are different surfaces; future docs should keep that deployment model clear.
